# Feature Specification: Chat Widget

**Feature Branch**: `004-chat-widget`
**Created**: 2026-03-15
**Status**: Draft
**Input**: User description: "F4 Chat widget — a React component embedded in the Docusaurus site that lets students ask questions about the textbook. The widget calls the F3 RAG backend POST /search to retrieve relevant passages, then calls the Gemini API (gemini-2.0-flash) to generate a grounded answer with citations. The widget appears as a floating chat button on every page."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Student Asks a Textbook Question (Priority: P1)

A student is reading any page of the Physical AI textbook and has a question. They click the floating chat button, type their question in natural language, and receive a grounded answer with numbered citations linking to specific textbook sections. The answer is generated by an LLM that has been given the top-ranked passages from the book as context.

**Why this priority**: Core value proposition — without this flow, the widget has no purpose. All other stories build on it.

**Independent Test**: Open any textbook page, click the chat button, type "What is the publisher-subscriber pattern in ROS 2?", and verify a coherent answer appears with at least one citation referencing a chapter and section.

**Acceptance Scenarios**:

1. **Given** the chat widget is open on any page, **When** the student types a question and submits, **Then** the widget displays a streamed or batch answer within 5 seconds, citing at least one textbook source.
2. **Given** a question is submitted, **When** the backend returns relevant passages, **Then** citations appear as labelled references (e.g., "[1] Chapter 3 — ROS 2 Architecture") beneath the answer.
3. **Given** the student submits an empty message, **When** they press send, **Then** the widget shows a validation hint and does not call any backend.
4. **Given** the RAG backend is unreachable, **When** the student submits a question, **Then** the widget shows a user-friendly error message and does not crash.

---

### User Story 2 - Widget Is Always Accessible (Priority: P2)

The chat button persists on every page of the Docusaurus site — homepage, chapter pages, and appendices — without disrupting reading. Students can open and close the chat panel at will. The widget does not cover main content when collapsed.

**Why this priority**: Discoverability and non-intrusiveness are prerequisites for adoption; the widget must be available everywhere without being annoying.

**Independent Test**: Navigate to three different pages (home, a module chapter, an appendix). On each, confirm the floating button is visible in the same corner. Open and close the panel. Confirm page content is fully readable when panel is closed.

**Acceptance Scenarios**:

1. **Given** any page in the Docusaurus site, **When** the page loads, **Then** the floating chat button is visible in the bottom-right corner.
2. **Given** the chat panel is closed, **When** the student clicks the button, **Then** the panel opens as an overlay without hiding the main content area.
3. **Given** the chat panel is open, **When** the student clicks the close/X button, **Then** the panel collapses back to the floating button.
4. **Given** the chat panel is open and the student navigates to another page, **When** the new page loads, **Then** the widget is still present (conversation may reset or persist — see Assumptions).

---

### User Story 3 - Cited Sources Are Traceable (Priority: P3)

When the widget displays an answer, each citation references a specific chapter and section of the textbook. Students can see which part of the book the answer came from. This builds trust and encourages deeper reading.

**Why this priority**: Citations differentiate this widget from generic AI chat — they anchor answers to the actual textbook content. Useful but separable from core Q&A flow.

**Independent Test**: Submit a question, receive an answer, inspect each citation. Verify that each citation shows at minimum the chapter title and section heading. Optionally verify that citation text matches the passage returned by POST /search.

**Acceptance Scenarios**:

1. **Given** an answer is displayed, **When** the student inspects citations, **Then** each citation shows chapter title and section heading from the source passage.
2. **Given** the RAG backend returns 3 passages, **When** the LLM uses all 3 as context, **Then** the answer body references them as [1], [2], [3] with a corresponding citation list.
3. **Given** the LLM answer only uses 2 of 3 passages, **When** the answer is displayed, **Then** only the used sources appear as citations (no phantom citations).

---

### Edge Cases

- What happens when the user submits a very long question (>500 chars)? Widget should accept it but may truncate before sending to backend.
- What happens when the Gemini API returns an error (quota exceeded, network timeout)? Show user-friendly error with retry option.
- What happens when POST /search returns 0 results? LLM should respond with "I couldn't find relevant information in the textbook" rather than hallucinating.
- What happens when the widget is opened on a mobile viewport? Panel should be full-width and scrollable.
- What happens if the student rapidly submits multiple questions? Only process one at a time — disable input while a response is in flight.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST display a floating chat button in the bottom-right corner of every Docusaurus page.
- **FR-002**: System MUST open a chat panel when the floating button is clicked, without navigating away from the current page.
- **FR-003**: System MUST call the RAG backend `POST /search` with the student's question to retrieve relevant passages before generating an answer.
- **FR-004**: System MUST call the Gemini API (`gemini-2.0-flash`) with the retrieved passages as context to generate a grounded answer.
- **FR-005**: System MUST display numbered citations below each answer, showing chapter title and section heading for each source passage used.
- **FR-006**: System MUST prevent submission of empty questions and show a validation message.
- **FR-007**: System MUST disable the input and show a loading indicator while a response is in flight.
- **FR-008**: System MUST display a user-friendly error message when the RAG backend or Gemini API is unreachable, without crashing.
- **FR-009**: System MUST render the chat panel as a floating overlay that does not obscure the main page content when open.
- **FR-010**: System MUST be implemented as a Docusaurus SwizzleComponent or plugin so it loads on every page without per-page changes.
- **FR-011**: System MUST NOT store the Gemini API key in client-side code; the key must be accessed via a server-side proxy or environment variable (build-time, with awareness of exposure risk).

### Key Entities

- **ChatMessage**: A single turn in the conversation — role (user/assistant), content text, optional list of citations, timestamp.
- **Citation**: Source passage metadata — chapter title, section heading, relevance score, optionally a short excerpt.
- **ConversationSession**: The in-memory list of ChatMessages for the current page visit. Resets on navigation (see Assumptions).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Students receive a grounded answer with at least one citation within 5 seconds of submitting a question on a typical broadband connection.
- **SC-002**: The floating button is visible on 100% of Docusaurus pages (home, chapters, appendices) without layout breakage.
- **SC-003**: Zero unhandled JavaScript exceptions when the RAG backend or Gemini API returns an error response.
- **SC-004**: The chat panel opens and closes without any visible shift or reflow in the main page content.
- **SC-005**: Answers for questions directly covered in the textbook include at least one citation that accurately references the source section.

## Assumptions

- **Conversation persistence**: Conversation resets on page navigation (simpler, no session storage needed). Can be enhanced in a future feature.
- **Gemini API key exposure**: In a Docusaurus static site, calling Gemini directly from the browser would expose the API key. The assumed mitigation is a thin server-side proxy endpoint (either an additional FastAPI route or a separate serverless function) that holds the key and forwards requests. The exact proxy design is left to planning.
- **Citation grounding**: The LLM is prompted to only cite sources from the provided context passages. Hallucinated citations are a known risk managed by prompt design.
- **top_k default**: Widget will use `top_k=5` for all searches (matching the RAG backend default).
- **Mobile support**: Widget must work on mobile viewports but does not need to be a native app experience.
- **No authentication**: Widget is publicly accessible — no login required to use the chat feature.

## Dependencies

- **F3 RAG Backend** (`003-rag-backend`): `POST /search` endpoint must be running and have book content ingested.
- **Docusaurus site** (`001-docusaurus-scaffold`): Site must be built and served.
- **Gemini API access**: `gemini-2.0-flash` model must be accessible (API key required).
